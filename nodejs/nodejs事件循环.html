# 事件循环

# 作用

# 事件循环机制解决了什么问题？

它是nodejs中异步的实现方式，它决定异步API的回调函数什么时候回到主线程中执行。

Nodejs是异步I/O模型（不阻塞等待I/O操作完成，而是继续执行其他指令，I/O操作完成通知CPU结果继续处理）。同步操作在主线程中执行，异步API在底层的C++维护的线程中执行。在JavaScript运行时，众多异步API回调函数什么时候回到主线程中调用呢，这就是时间循环机制做的事情。

# nodejs事件循环完整过程：

  1. 执行同步代码，遇到异步代码异步API将其回调函数放入对应的队列；
  2. 同步代码执行完成，执行微任务队列的回调函数；
  3. 微任务执行完成，从Timers阶段开始执行宏任务，执行完当前阶段队列中的所有宏任务回调函数后执行微任务，执行完一个宏任务再检查微任务队列执行微任务；
  4. 微任务执行完成，进入下一个阶段的宏任务，执行任务队列内的一个宏任务，再去执行微任务，循环往复走完6个阶段；

删除线内容说明，删除部分为node10之前的事件循环实现，node11开始事件循环和浏览器更为接近。

# 宏任务和微任务

异步操作分为宏任务和微任务，宏任务的回调函数放在宏任务队列中，微任务的回调函数放在微任务队列中。

  • 宏任务（macro task）分为6个阶段（6个队列），下文详细内容；
  • 微任务（micro task）包括：process.nextTick、promise相关的几个；
  • 微任务执行优先级高于宏任务；
  • 宏任务之间没有优先级，执行顺序按照队列和事件循环的阶段；
  • 微任务中process.nextTick优先级高于其他微任务，其他任务按队列顺序执行

# 宏任务的6个阶段

  1. Timers：存储执行定时器setTimeout,setInterval的回调函数队列；
  2. Pending Callbacks：存储执行与操作系统有关的回调函数。比如启动服务器端应用时监听端口回调函数；
  3. Idle，prepare：系统内部使用；（不需要关注）
  4. IO Poll：存储执行IO操作的回调函数队列；例如文件读写、数据库操作的回调函数
  5. Check：存储执行setImmediate的回调函数；
  6. Close Callbacks：存储执行与关闭事件的回调函数；例如数据库关闭、socket连接关闭等的回调函数。

每个阶段执行前都会检查微任务队列是否为空，不为空，则先执行微任务队列中的回调函数。

# 常见问题

例1：

setTimeout(() => console.log('1'), 0)
setImmediate(() => console.log('2))
// 打印 2 1或者 1 2，没有确切的先后顺序

解释：

setTimeout(,0)nodejs中会被转成setTimeout(,1)，回调函数加入timer队列的时间间隔最小是1ms。

  • 2 1的情况：如果系统进入时间循环的时间点还不到1ms，那么Timers阶段队列中还没有回调函数，顺延到Check Callbacks阶段队列中的回调函数就被执行，打印2，下一圈循环到Timers队列，setTimeout的回调函数才执行打印1。
  • 1 2的情况：如果系统进入时间循环的时间点还超过了1ms，那么Timers阶段队列中的回调函数执行打印1，顺延到Check Callbacks阶段队列中的回调函数就被执行，打印2。

例2：

const fs = require('fs')

fs.readFile('test.js', () => {
    setTimeout(() => console.log('1'), 0)
    setImmediate(() => console.log('2))
})
// 确切的输出：2 1

解释：

readFile回调函数会被放在IO Poll阶段的队列，下个阶段是Check，执行setImmediate对应的回调函数，打印2，事件循环进入下一圈，进入到Timer阶段，setTimeout的回调函数执行，打印1。
