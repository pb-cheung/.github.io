What the heck is the event loop anyway?(Philip Roberts) (opens new window)

# JavaScript中的事件是如何工作的

对JS常见的解释是这样的：它是一个单线程、非阻塞、异步、解释性的脚本语言。它由调用栈、事件循环、回调队列和一些API等组成。

# 浏览器引擎只包含堆和调用栈

V8引擎中有堆（Heap）和调用栈（Stack），没有其他的东西（事件循环、回调队列、APIS）

V8源码中搜索不到DOM、HTTP请求（XMLHttpRequest）相关的代码。

V8引擎提供运行环境，js运行所需其他由浏览器环境提供。

# 回调栈

one thread == one call stack == one thing at a time

单线程 == 一个回调栈 == 某个时间点只做一件事

so the call stack is basically ‑‑ it's a data structure which records basically where in the program we are, if we step into a function, we put something on to the stack, if we return from a function, we pop off the top of the stack that's all the stack can do

调用栈是一个记录当前程序所在位置的数据结构，如果当前进入了某个函数，某个函数就会被放在栈里面。如果当前离开了某个函数，这个函数就会被弹出栈。

# 阻塞——什么使程序变慢

Things which are slow and on that stack are what are blocking means

在栈里表现很慢的东西都叫阻塞。

举例：

单线程语言运行多个阻塞任务的情况。前三行是三个HTTP请求，耗时较久（占用栈时间长），本例中为阻塞。 后三行console.log运行需要等很久时间，等前三行的“阻塞”执行完毕。

# 阻塞带来的问题

因为我们的JavaScript代码是运行在浏览器中的，“阻塞”会影响用户交互影响体验。

“阻塞”会导致浏览器无法响应（无法响应点击等事件、无法渲染）。直到栈中的“阻塞”执行完毕。

# 事件循环

JS中并发/异步的实现原理：遇到“阻塞”（DOM事件、网络请求），不会等着待其执行；而是将其挂起，等到“阻塞”返回结果，将其回调函数放入回调队列中，当调用栈中的所有任务都执行完毕，从回调队列头部取出一个回调函数执行。
